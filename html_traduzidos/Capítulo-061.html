O que você precisa para construir a IU: Navigation and the InheritedWidget

Antes de realmente construirmos a IU para este aplicativo, vamos discutir duas coisas importantes: como navegar entre várias visualizações no Flutter e como usar InheritedWidget s. Esta será uma introdução geral às opções que o Flutter oferece para navegação; usaremos apenas alguns deles no aplicativo. Se você preferir avançar para navegação mais avançado, dê uma olhada Criar o UI básico do App e depois voltar aqui quando você quiser saber mais sobre como implementar a navegação em aplicativos vibração.

Navegação em Flutter

O Flutter oferece várias opções para navegar entre as visualizações:

  • navegação push / pop , útil quando há uma página inicial e muitas páginas acessíveis a partir dela, potencialmente usando um menu de gaveta .

  • Navegação página a página usando PageView e PageController , útil quando há um número limitado de páginas que podem ser consideradas no mesmo nível.

Usando um menu de navegação do lado da gaveta

Um elemento de navegação comumente visto é o Drawer , [39] que é o menu, visto em muitos aplicativos, que abre deslizando para a direita começando da borda esquerda da tela ou tocando no botão comumente visto com três linhas horizontais verticalmente empilhadas.

Como muitos itens que são comumente usados ​​e colocados fora do corpo do aplicativo, uma gaveta é adicionada ao aplicativo usando a opção de gaveta do Andaime e definindo-o como um objeto Gaveta :

Para este exemplo, usaremos um ListView .

Navegação página a página

A navegação deslizante página a página pode ser alcançada usando dois tipos de visualização: a PageView e a TabBarView , a diferença é que a primeira só permite a navegação deslizando, enquanto a segunda é usada junto com uma TabBar abaixo da AppBar .

Se, em vez disso, você quiser apenas algo como o aplicativo do YouTube: ser capaz de alternar entre as páginas usando um menu na parte inferior da tela, o caminho a percorrer é com uma BottomNavigationBar .

BottomNavigationBar

Um BottomNavigationBar funciona muito como um Drawer , mas está sempre visível na parte inferior, então você não precisa usar o Navigator.pop .

Mais especificamente, você adicioná-lo ao andaime ‘s bottomNavigationBar opção, e você usá-lo como este:

​  retorno andaime (

​  currentIndex: index,
​  Itens: [
​  BottomNavigationBarItem (
​  ícone: ícone (...),
​  título: Texto (...),
​  ),
​  BottomNavigationBarItem (
​  ícone: ícone (...),
​  título: Texto (...),
​  ),
​  // potencialmente mais itens de navegação ...
​  ],
​  onTap: (página) {
​  setState (() {
​  interruptor (página) {
​  caso 0:
​  // Coisas que fazer se o usuário seleciona o primeiro item
​  índice = 0;
​  quebrar ;
​  caso 1:
​  // Coisas que fazer se o usuário seleciona o segundo item

Tem três opções principais:

  • Os itens , que são as opções de navegação a serem fornecidas ao usuário.

  • O retorno de chamada onTap , que leva um número inteiro indicando qual opção foi tocada pelo usuário.

O PageView é comparável a um ListView horizontal , mas só rola página por página.

InheritedWidgets

O DefaultTabController é um tipo especial de widget: um InheritedWidget , que é usado para compartilhar informações de estado entre widgets Flutter, armazenando-os em um widget que fica no topo da árvore de widgets e permitindo acesso direto a seus campos e métodos por seus filhos , para que eles não precisem ser passados ​​para a árvore usando argumentos do construtor.

Eles geralmente são acessados ​​usando a sintaxe Widget.of (context) .fieldName que usamos anteriormente para obter dados de tema do Tema e para mostrar barras de lanche no Scaffold . Isso ocorre porque tanto o Tema quanto o Scaffold são, na verdade, InheritedWidget s.

Um InheritedWidget é definido da seguinte maneira:

​  classe Exemplo estende InheritedWidget {
​  Exemplo ({criança, este .field}):
​  Super (child: criança);
​  definitiva Cadeia campo;
​ 
​  @sobrepor
​  boleano updateShouldNotify (Exemplo de idade) => verdadeira ;
​  }

updateShouldNotify é usado para determinar se os widgets filhos devem ser reconstruídos quando o InheritedWidget é reconstruído pela estrutura.

Isso geralmente será verdade , mas aí você também pode definir um conjunto de condições para determinar se, por exemplo, há alterações nos campos do widget e apenas reconstruir os filhos nesse caso.

Em seu estado atual, Example.of (contexto) .fieldName de uma criança de Exemplo não iria funcionar desde há realmente nenhum de método.

Embora seja muito comum, na verdade não está embutido no framework: por padrão, você acessa o pai InheritedWidget s usando context.inheritFromWidgetOfExactType (Exemplo) .

Na verdade, um método muito comum estático (explicado em membros estático ) definido dentro InheritedWidget s é o de modo que, no caso de um InheritedWidget chamado Exemplo , pode ser definida da seguinte forma:

​  estática Exemplo de (contexto BuildContext) =>
​  context.inheritFromWidgetOfExactType (exemplo);

para que toda a nossa definição de classe Exemplo se torne:

​  classe Exemplo estende InheritedWidget {
​  Exemplo ({criança, este .field}): Super (criança: criança);
​  definitiva Cadeia campo;
​ 
​  @sobrepor
​  bool updateShouldNotify (Exemplo de idade) => old.field = campo!;
​ 
​  estática Exemplo de (contexto BuildContext) =>
​  context.inheritFromWidgetOfExactType (exemplo);
​  }
