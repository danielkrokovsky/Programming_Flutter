Construindo o aplicativo de bate-papo

Agora que sabemos como usar o Firebase para o que precisamos (autenticação e banco de dados) e temos algumas noções de IU mais avançadas, podemos começar a construir o aplicativo de chat que prometi que trabalharíamos.

Planejando a IU do nosso aplicativo

Um aplicativo de bate-papo precisa que o usuário execute várias ações ao iniciar o aplicativo pela primeira vez:

  • Precisamos que o usuário insira um endereço de e-mail e senha e faça login em uma conta existente ou crie uma nova se ainda não tiver uma. Nesse momento, o usuário também deve receber um e-mail solicitando a verificação de sua conta de usuário, mas não faremos distinções entre usuários verificados e não verificados neste momento.

  • Precisamos que o usuário escolha um nome de exibição a ser usado para se identificar para outros usuários no aplicativo.

  • Se o usuário fez tudo isso, ele deve ir para a página que mostra todas as mensagens enviadas por outros usuários.

Construindo a IU

Vamos construir a IU criando cada peça na ordem em que o usuário irá interagir com ela. Vamos começar com a página de Inscrição / Inscrição!

Adicionar uma página de perfil: armazenar dados do usuário no Cloud Firestore

Existem alguns problemas com nosso aplicativo: dois usuários com o mesmo nome de exibição criariam confusão, já que estamos apenas armazenando o nome de exibição de um usuário no banco de dados, e não seu ID, então nosso aplicativo pensaria que um usuário é o remetente de uma mensagem mesmo se o usuário que enviou a mensagem for um usuário completamente diferente que por acaso tenha o mesmo nome de exibição.

Além disso, o aplicativo realmente não fornece nenhuma maneira de distinguir entre esses dois usuários, o que seria uma situação confusa para todos. Um bom começo seria fornecer aos usuários uma forma de definir uma string curta para personalizar seu perfil (geralmente chamada de status ou biografia) e uma forma de entrar em contato com outro usuário do aplicativo de bate-papo por e-mail, uma vez que já estamos solicitando isso em formação.

Criação de uma coleção de dados do usuário

Vamos criar outra coleção! Vamos chamá-lo de usuários e será estruturado da seguinte forma:

  • O uid do usuário será o ID do documento, para facilitar o acesso ao documento correspondente a cada usuário.

  • O displayName do usuário será uma string chamada displayName .

  • O endereço de e-mail do usuário será outra string chamada e-mail .

  • A biografia do usuário será armazenada como uma string chamada bio .

Alterando os documentos e o aplicativo

Precisamos criar entradas para a coleta de dados do usuário para cada usuário existente de nosso aplicativo e alterar cada entrada na coleção de mensagens e substituir o nome de exibição pelo ID do usuário. Então, vamos nos preocupar em fazer o aplicativo funcionar com as mensagens alteradas no banco de dados, fazê-lo exibir o nome de exibição obtido do Firestore e, em seguida, trabalharemos na construção de uma página de perfil.

Alterando os Dados

A criação da coleção dos usuários deve ser feita manualmente e geralmente é uma boa ideia ter uma coleção contendo dados do usuário: é assim que é na maioria dos bancos de dados “tradicionais” e vale a pena replicá-la no Firebase, exceto pelo fato de que não não precisa (e não deveria, e especialmente não em texto simples) para armazenar senhas no banco de dados que podemos acessar.

Vamos começar a lidar com a mudança de dados: alterar cada mensagem no banco de dados e substituir os valores é fácil o suficiente quando não há muitas mensagens, pois podemos apenas alterá-las manualmente no console do Firebase, mas com bancos de dados maiores você teria para escrever algum código que (usando as interfaces que aprendemos neste capítulo) busque as mensagens, encontre o usuário com o nome de exibição fornecido e substitua o nome de exibição pelo ID do usuário.

Fazendo transições suaves

Durante o desenvolvimento inicial (e no nosso caso), um aplicativo não tem usuários, então podemos fazer alterações no banco de dados que impedem uma versão anterior do aplicativo de funcionar a qualquer momento, mas não é o caso quando um aplicativo já está no mercado e uma mudança significativa é feita na maneira como os dados são armazenados.

Certamente, uma das primeiras etapas seria informar ao usuário de uma versão anterior que sua versão do aplicativo não funciona mais em vez de apenas travar o aplicativo, mas dar ao usuário uma janela de tempo para realizar a atualização também seria algo bom, e isso seria feito, no nosso caso, armazenando o ID do usuário como uma propriedade adicional de uma mensagem, em vez de substituir o usado anteriormente e, em seguida, removendo a propriedade do nome de exibição (agora inútil) depois de dar algum tempo para os usuários atualizem, mostrando aos usuários da versão anterior um aviso de que não devem mais usar aquela versão.

Além disso, para outros usuários que atualizaram sua versão do aplicativo antes que muitos tivessem a chance de definir uma biografia, você teria que definir a biografia de cada usuário existente para algum valor padrão (que poderia ser um "" em branco ou algo diferente, como aqueles que outros aplicativos de bate-papo usam).

Tornando o aplicativo mais seguro, bloqueando o acesso ao Firestore

Como implementamos a autenticação, não devemos manter as regras do banco de dados do Cloud Firestore tão abertas como estão agora porque, embora permitamos que qualquer pessoa se inscreva, devemos tentar restringir o acesso ao mínimo necessário para que o aplicativo funcione.

O que vamos fazer

Mais especificamente, precisamos fazer o seguinte:

  • Permitir apenas que usuários autenticados com um determinado uid adicionem ou editem um usuário com aquele uid à coleção de usuários, permitindo que todos leiam os dados do usuário, mas não permitindo que ninguém exclua usuários (no momento não estamos oferecendo essa opção).

  • Permitir apenas que usuários autenticados com um determinado uid adicionem uma mensagem que declare ser desse uid , permitindo que todos acessem as mensagens, mas não permitindo que ninguém as edite ou exclua (pelo mesmo motivo: não estamos oferecendo essa opção aos usuários em o momento).

Implementando a Restrição

Tudo isso é obtido pela seguinte configuração do Cloud Firestore (a ser inserido em Banco de dados -> Regras no console do Firebase):

​  service cloud.firestore {
​  corresponder / bancos de dados / {banco de dados} / documentos {
​  corresponder / Usuários / {userId} {
​  permitir criar, atualizar: if request.auth.uid == userId;
​  permitir leitura: if request.auth! = null;
​  }
​ 
​  corresponder / Mensagens / {messageId} {
​  permitir a criação: if request.auth.uid == request.resource.data.from;
​  permitir leitura: if request.auth! = null;
​  }
​  }
​  }

Documentos correspondentes

corresponder / bancos de dados / {banco de dados} / documentos torna o caminho raiz de tudo nas chaves após essa instrução no nível da coleção, o que significa que, por exemplo, / Usuários / ABCD é o documento da coleção Usuários que tem ID ABCD .

Dentro disso, temos outras duas instruções de correspondência :

  • match / Users / {userId} , onde vamos regular o acesso à coleção de usuários .

  • match / Messages / {message} , onde vamos nos preocupar com o acesso à coleção de mensagens .

permitindo acesso

As declarações de permissão são usadas da seguinte maneira:

​  permitir operação: se condição;

A operação pode ser apenas leitura ou gravação , ou um aspecto mais específico da gravação , como criar , atualizar e excluir . Também pode ser uma lista de operações separadas por vírgulas.

A condição pode ser apenas verdadeira ou falsa (como fizemos no início), mas deve levar em consideração que documento o usuário está tentando acessar e quem é o usuário (se está autenticado, qual é seu UID, etc. .).

Em nosso caso, para cada condição, podemos usar três dados: o que quer que esteja entre chaves no caminho de qualquer instrução de correspondência que o inclua (por exemplo, no caso da instrução de correspondência de Usuários , {userId} , que é o caminho do documento do usuário, ou seja, seu ID), a interface de solicitação e a interface de recurso .

Vamos começar com o último: as interfaces de recursos contêm dados sobre o recurso que estamos tentando ler, atualizar ou excluir. Por exemplo, se queríamos para acessar a partir de campo de uma mensagem, que iria escrever resource.data.from .

Muito semelhante a resource é request.resource , que são os dados de entrada da solicitação. Por exemplo, se a solicitação está tentando adicionar um usuário, acessaríamos a biografia desse usuário com request.resource.data.bio . Outra propriedade importante da interface de solicitação é request.auth , que nos informa se o usuário que está tentando acessar o Firestore está autenticado e quais são suas credenciais. Por exemplo, se o usuário não for autenticado, request.auth será igual a nulo ; se, em vez disso, o usuário for autenticado, request.auth.uid retornará o UID do usuário que está tentando acessar o Firestore.

Combinar todos esses aspectos e nossos requisitos nos dá o seguinte para os usuários :

​  corresponder / Usuários / {userId} {
​  permitir criar, atualizar: if request.auth.uid == userId;
​  permitir leitura: if request.auth! = null;
​  }

e o seguinte para as Mensagens :

​  corresponder / Mensagens / {messageId} {
​  permitir a criação: if request.auth.uid == request.resource.data.from;
​  permitir leitura: if request.auth! = null;
​  }

Algumas palavras sobre como tornar os dados no Firestore mais seguros

Agora os dados estão inacessíveis para usuários anônimos e, no caso de um chat público, tudo bem. Mas e os aplicativos de chat individual ou em grupo? Por um lado, você precisaria separar as mensagens das diferentes conversas, o que seria bastante simples, mas uma questão realmente importante é a da privacidade e da confidencialidade: as mensagens seriam armazenadas em texto simples no banco de dados, permitindo que aqueles que executam o aplicativo espionar conversas, que seriam todas reveladas no caso de um incidente de segurança. Essa é a razão pela qual a maioria dos aplicativos de chat individual implementa algum tipo de criptografia, armazenando no banco de dados mensagens criptografadas que só podem ser criptografadas usando chaves armazenadas no dispositivo de cada usuário.

Redefinição de senha e verificação de endereços de e-mail

Para fins de nosso aplicativo de bate-papo, realmente não precisamos diferenciar os usuários que verificaram seus endereços de e-mail e aqueles que não verificaram, já que não estamos nem oferecendo redefinição de senha por e-mail (o que é feito executando _auth.sendPasswordResetEmail ) No entanto, você pode optar por fazer com que aqueles que ainda não verificaram seu e-mail apareçam na lista de mensagens de forma diferente (alterando o texto do nome de exibição para uma cor diferente, por exemplo, ou mostrando um ícone de aprovação ao lado de usuários com endereços de e-mail verificados ), mas não é realmente necessário.

No caso de outros aplicativos, a capacidade de fazer tudo isso sem ter que depender de nada além do Firebase e de forma tão simples é certamente um bom motivo para estar interessado no Firebase como uma ferramenta para ajudar a criar aplicativos que, de outra forma, teriam que dependem de servidores e código de back-end separado para funcionar.
