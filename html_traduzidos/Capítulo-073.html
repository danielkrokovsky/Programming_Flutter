Lançamento e captura de exceções

Já usamos exceções em alguns dos capítulos anteriores, mas examinaremos a implementação do Flutter com mais detalhes nesta seção.

Exceções são usadas em um pedaço de código como uma forma de sinalizar errado Isso é algo que se foi, a função atual (ou Funções anônimas e Closures ) execução 's é interrompido e a função de chamada é suposto para lidar com isso por captura-lo.

Lançando uma exceção

No Dart, as exceções são lançadas usando a palavra-chave throw (em Python, você chamaria isso de gerar uma exceção, caso nunca tenha conseguido ver como é o mundo fora do Python). Você pode criar sua própria exceção, criando uma classe que escrevendo uma classe que implementa uma interface da Exceção de erro de classe:

​  classe MyOwnPersonalException implementos Exception {
​  MyOwnPersonalException ( este .myOwnPersonalMember);
​ 
​  Cordas myOwnPersonalMember;
​  }
​ 
​  duplo myPersonalDivision ( duplo um, duplo b) {
​  se (b == 0) {
​  jogue MyOwnPersonalException ( "Falha ao dividir um número por 0." );
​  }
​  voltar a / b;
​  }

Capturando uma exceção

Sempre que chamamos uma função ou usamos uma operação que pode falhar em certas circunstâncias que não estamos verificando, devemos capturar a exceção:

​  vazio responsibleFunction () {
​  int a, b;
​  // ...
​  b = 0;
​  tente {
​  int d = myPersonalDivision (a, b);
​  // ...
​  }
​  prendedor (exceção) {
​  // lidar com a exceção
​  }
​  // ...
​  }

Você pode substituir exceção entre os parênteses de catch com o que quiser (geralmente é apenas e ). Ele contém o nome da exceção que foi lançada. Você pode imprimi-lo no console de depuração usando imprimir (exceção) . Além disso, você pode transmitir um rastreamento de pilha da origem do erro, que também pode ser impresso.

Por exemplo, o seguinte:

​  tente {
​  int .parse ( "vibração" );
​  } Captura (e, stacktrace) {
​  imprimir (e);
​  imprimir (rastreamento de pilha);
​  }

em SomeException catch ()

E se essa função chamada tivesse lançado outro tipo de exceção, que não tinha myOwnPersonalMember ou talvez você precisasse lidar com diferentes exceções de maneiras diferentes? Isso é o que nos é para.

Por exemplo, digamos que temos uma função que pega dois números em um formato de string, os analisa e executa a divisão de inteiros neles. Isso significa que dois tipos diferentes de exceções podem ser lançados durante sua execução: um FormatException se não puder analisar os números ou uma exceção IntegerDivisionByZeroException se tentarmos dividir por zero.

Por padrão, o Dart executa a divisão de ponto flutuante e retorna um número duplo . Para realizar a divisão inteira, precisamos usar o operador ~ / em vez de apenas / . Isso significa que a definição da função será a seguinte:

​  int jsonDivide ( Cadeia A, corda b) =>
​  int .parse (a) ~ / int .parse (b);

Visto assim, não é de se admirar por que uma função como esta é um bom exemplo: é muito curta, mas poderia lançar dois tipos diferentes de exceções: por exemplo, chamá-la como jsonDivide ("5", "0") retornará um IntegerDivisionByZeroException , enquanto chamá-lo como jsonDivide ("Flutter", "10") retornará um FormatException . Vamos fingir que precisamos dizer ao usuário para corrigir os dados incorretos inseridos, então precisamos lidar com cada exceção de maneira diferente, o que não é difícil de fazer no Dart:

​  tente {
​  jsonDivide (firstString, secondString);
​  }
​  on IntegerDivisionByZeroException {
​  // dizer que o usuário não dividir por 0
​  }
​  em FormatException {
​  // dizer que o usuário insira números válidos
​  }

De modo que cada um dos dois em blocos será executada somente quando a exceção correspondente é lançada.

Uma sintaxe alternativa para blocos on é em ExceptionName catch (e) {} , que passará a exceção para o bloco, assim como o que acontece em blocos catch regulares .

Você pode, opcionalmente, adicionar um bloco catch após os blocos on para tratar quaisquer exceções que não sejam tratadas pelos blocos on .

Finalmente

Ao usar uma construção try - catch regular sem lançar novas exceções no processo, a execução continuará executando as instruções colocadas após o final do bloco catch , mas se você estiver usando apenas blocos em vez de blocos catch genéricos ou lançando novas exceções dentro dos blocos catch , a execução será interrompida e essas instruções não serão executadas. Se, em vez disso, você quiser executar algumas instruções independentemente de qualquer coisa que aconteça no bloco try e no bloco catch e on , você usaria um bloco finally :

​  tente {
​  // código que poderia falhar
​  }
​  em SomeException {
​  // SomeExceptin punho
​  }
​  finalmente {
​  // código que certamente irá ser executado
​  }
